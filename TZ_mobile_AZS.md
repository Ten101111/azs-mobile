# ТЗ: мобильное приложение для просмотра объектов АЗС

## 1. Назначение продукта

Мобильное приложение предназначено для оперативного просмотра информации по объектам АЗС и связанным сервисным/управленческим атрибутам. Пользователь должен быстро найти объект, увидеть его текущее состояние, местоположение, основные характеристики, ответственных лиц и признаки, важные для дальнейшей аналитики.

На первом этапе приложение работает как мобильный справочник объектов с картой и карточкой АЗС. На следующих этапах добавляются аналитические показатели, сравнение объектов, мониторинг изменений и управленческие дашборды.

## 2. Проанализированные источники данных

В рабочей папке находятся два файла:

- `data.csv`: 2 424 строки, 57 полей. Поле `Код КССС` заполнено и уникально для всех строк, поэтому подходит как ключ объекта в этой выгрузке.
- `cls_2026_05_AZS.xlsx`: основной лист `cls_AZS` содержит 2 770 строк и 140 полей. Поле `КССС_union` заполнено и уникально для всех строк, поэтому его стоит использовать как основной ключ в целевой модели.

Дополнительные листы Excel:

- `Лист3`: список КССС для исключений.
- `франчайзинг+комиссия`: список объектов с признаком исключения.
- `Выход с реконструкции 2025`: объекты, выходившие из реконструкции в 2025 году.
- `Локация_new`: предложения по корректировке классификации локации и типа окружающей местности.
- `cls_знаковые проекты_pq`: признаки знаковых проектов, включая М-11, М-12, городские и трассовые флагманы.
- `Лист2`: классификация LukCafe.

Важные наблюдения по данным:

- В `data.csv` 2 332 объекта имеют состояние `Действующая`.
- В `cls_AZS` статусы шире: `Действующая`, `CODO`, `Передана в другое НПО`, `Передана в аренду`, `Оптимизация`, `Консервация`, `Франчайзинг+комиссия` и другие.
- В данных есть координаты почти по всем объектам, но обнаружены некорректные/нулевые координаты. Такие объекты нельзя показывать на карте как реальные точки без проверки.
- Есть поля с высоким процентом пропусков: свободная электромощность, дополнительные сервисы, даты/статусы передач, некоторые показатели бизнес-плана.
- Есть неоднородные значения справочников: например `да`/`Да`, `нет`/`Нет`, `город`/`Город`. Перед загрузкой в приложение нужна нормализация.

## 3. Целевая аудитория и роли

### 3.1. Роли MVP

1. Руководитель / региональный управляющий
   - смотрит список объектов по региону/НПО;
   - проверяет статус, формат, сервисы и ответственных;
   - быстро переходит к паспорту объекта.

2. Территориальный менеджер / агент
   - находит свои объекты;
   - видит адрес, контакты, состояние, инфраструктуру;
   - использует карту и маршрут.

3. Аналитик / центральный офис
   - проверяет полноту данных;
   - смотрит распределения по форматам, статусам, сервисам;
   - готовит будущие аналитические сценарии.

### 3.2. Роли следующих этапов

- Администратор данных: управляет загрузками, справочниками, качеством данных и правами доступа.
- Руководитель направления сервисов: анализирует магазины, кафе, санузлы, LukCafe и сервисные кластеры.
- Руководитель проектов: отслеживает реконструкции, знаковые проекты и изменения статусов.

## 4. MVP: функциональные требования

### 4.1. Главный экран

Главный экран должен открываться сразу в рабочий режим, без промо-страницы.

Элементы:

- строка поиска по номеру АЗС, КССС, адресу, региону, городу и названию;
- переключатель режимов `Список` / `Карта`;
- быстрые фильтры: НПО, регион/субъект, статус, тип АЗС, формат, локация, наличие магазина, наличие кафе, наличие санузла;
- счетчик найденных объектов;
- индикатор даты последнего обновления данных.

### 4.2. Список объектов

Каждая строка списка должна показывать:

- номер АЗС;
- КССС;
- адрес или город/субъект;
- статус;
- тип объекта;
- формат;
- НПО;
- бейджи: магазин, кафе, санузел, LukCafe, знаковый проект, реконструкция 2025, агентская схема.

Список должен поддерживать:

- сортировку по номеру АЗС, региону, статусу, формату, дате изменения;
- сохранение выбранных фильтров;
- быстрый переход в карточку объекта.

### 4.3. Карта

Карта должна показывать объекты с валидными координатами.

Требования:

- маркеры с цветом по статусу объекта;
- кластеризация маркеров при большом масштабе;
- фильтры карты синхронизированы со списком;
- при нажатии на маркер отображается краткая карточка: номер, КССС, адрес, статус, формат, сервисы;
- кнопка открытия полной карточки;
- кнопка построения маршрута через системное приложение карт.

Объекты с отсутствующими или нулевыми координатами показывать в отдельном списке `Без координат`.

### 4.4. Карточка объекта

Карточка объекта должна быть разделена на блоки.

#### Основное

- КССС / `КССС_union`;
- номер АЗС;
- название АЗС, если заполнено;
- тип АЗС;
- статус;
- НПО;
- субъект / регион;
- город;
- адрес;
- координаты;
- ссылка на паспорт АЗС, если доступна.

#### Ответственные и контакты

- региональный управляющий;
- территориальный менеджер / агент;
- менеджер АЗС;
- старший оператор;
- телефоны с возможностью позвонить одним нажатием.

#### Формат и классификация

- формат;
- формат level 2;
- формат Minale;
- локация;
- тип окружающей местности;
- признак действующей АЗС;
- признак MD;
- признаки передачи в Ликард, Тебойл, агентскую схему.

#### Сервисы

- торговый зал / магазин;
- площадь торгового зала;
- режим работы торгового зала;
- наличие кафе;
- кластер по сервису;
- формат LukCafe L1/L2;
- наличие санузла;
- тип оплаты;
- пост подкачки шин;
- дополнительные сервисы.

#### Инфраструктура

- количество ТРК;
- количество постов ТРК;
- количество постов ТРК без навеса;
- площадь операторной;
- площадь земельного участка;
- водоснабжение;
- бытовая канализация;
- электромощность максимальная;
- свободная электромощность;
- наличие стационарной ДЭС/ДГУ.

#### Трасса и окружение

- локация: город, трасса, прочее;
- федеральная трасса;
- учетный номер трассы;
- наименование трассы;
- сторона трассы;
- плотность населения;
- тип окружающей местности;
- наличие основного конкурента;
- наименование конкурента;
- расстояние до конкурента.

#### История и события

- год/дата ввода в эксплуатацию;
- год/дата последней реконструкции;
- статус начала/окончания, если есть;
- дата добавления/изменения;
- комментарии;
- признак выхода из реконструкции 2025;
- признаки знаковых проектов.

### 4.5. Избранное и недавние

Пользователь должен иметь возможность:

- добавить объект в избранное;
- открыть список недавно просмотренных объектов;
- быстро вернуться к последним фильтрам и карте.

### 4.6. Качество данных

В MVP необходимо показывать технические предупреждения:

- отсутствуют координаты;
- координаты равны `0`;
- не заполнен адрес;
- не заполнен статус;
- не заполнен ответственный;
- конфликтующие или нестандартизированные значения справочников.

Предупреждения должны быть видны в карточке объекта, но не должны мешать обычному просмотру.

## 5. Аналитика: следующие этапы

### 5.1. Дашборд по сети

Показатели:

- всего объектов;
- действующие объекты;
- объекты по статусам;
- объекты по НПО;
- объекты по регионам;
- распределение по типам АЗС;
- распределение по форматам;
- доля объектов с магазином, кафе, санузлом;
- доля объектов с валидными координатами;
- количество объектов с признаками LukCafe;
- количество знаковых проектов;
- количество объектов в реконструкции / вышедших из реконструкции.

### 5.2. Аналитика по региону / НПО

Фильтруемый экран с KPI:

- количество объектов;
- структура по статусам;
- структура по форматам;
- сервисное покрытие: магазин, кафе, санузел, LukCafe;
- проблемные объекты по качеству данных;
- топ регионов/городов по количеству объектов;
- объекты без координат;
- объекты с устаревшей датой изменения.

### 5.3. Аналитика по объекту

В карточку объекта можно добавить:

- план/факт по бизнес-показателям;
- динамику продаж;
- НТУ;
- т/сут;
- сравнение с объектами того же региона, формата и локации;
- расстояние до конкурентов;
- события реконструкций и изменений статуса.

## 6. Модель данных MVP

### 6.1. Основная сущность `Station`

Обязательные поля:

- `id`: внутренний идентификатор;
- `ksss_union`: основной ключ из `cls_AZS`;
- `ksss_code`: код КССС из CSV/источников;
- `station_number`: номер АЗС;
- `station_name`;
- `station_type`;
- `status`;
- `npo`;
- `region`;
- `subject`;
- `city`;
- `address`;
- `latitude`;
- `longitude`;
- `last_updated_at`.

### 6.2. Связанные сущности

- `StationContacts`: ответственные лица и телефоны.
- `StationServices`: магазин, кафе, LukCafe, санузел, сервисный кластер.
- `StationInfrastructure`: ТРК, посты, площади, электромощность, водоснабжение, канализация.
- `StationRoadContext`: трасса, сторона трассы, локация, окружение, конкуренты.
- `StationEvents`: реконструкции, передачи, статусы начала/конца, комментарии.
- `StationFlags`: знаковые проекты, исключения, агентская схема, Ликард, Тебойл, MD.
- `DataQualityIssue`: ошибки и предупреждения по данным.

## 7. Загрузка и обновление данных

### 7.1. MVP-источник

На первом этапе источник данных: загружаемый Excel/CSV-файл.

Процесс:

1. Администратор загружает новый файл.
2. Backend парсит листы.
3. Данные нормализуются.
4. Выполняются проверки качества.
5. Создается версия датасета.
6. Мобильное приложение получает актуальную версию через API.

### 7.2. Нормализация

Нужно нормализовать:

- статусы;
- типы АЗС;
- форматы;
- значения `да/нет`;
- названия регионов/субъектов;
- телефоны;
- числовые поля с запятой как десятичным разделителем;
- даты;
- координаты;
- дублирующиеся группы колонок в `cls_AZS`.

### 7.3. Версионирование

Каждая загрузка должна иметь:

- дату загрузки;
- имя исходного файла;
- количество строк;
- количество успешно загруженных объектов;
- количество ошибок;
- количество предупреждений;
- пользователя, выполнившего загрузку.

## 8. API MVP

Минимальный набор методов:

- `GET /stations`: список объектов с поиском, фильтрами, сортировкой и пагинацией;
- `GET /stations/{id}`: полная карточка объекта;
- `GET /stations/map`: облегченная выборка для карты;
- `GET /stations/{id}/quality`: предупреждения по качеству данных;
- `GET /dictionaries`: справочники для фильтров;
- `GET /dataset/current`: текущая версия данных;
- `POST /imports`: загрузка нового файла, только для администратора;
- `GET /imports/{id}`: статус обработки загрузки.

## 9. Нефункциональные требования

- Платформы: iOS и Android.
- Приложение должно работать стабильно на мобильной сети.
- Поиск по локально закешированным объектам должен быть быстрым.
- Карта должна поддерживать кластеризацию, чтобы не перегружать экран.
- Основные данные по ранее открытым объектам должны быть доступны офлайн.
- Чувствительные контакты и внутренние ссылки должны быть доступны только авторизованным пользователям.
- Все API должны работать по HTTPS.
- Доступ к ролям и данным должен управляться на сервере.

## 10. UX-требования

- Первый экран приложения должен быть рабочим: список/карта объектов.
- Карточка объекта должна быть компактной, с раскрываемыми секциями.
- Важные признаки показывать бейджами, а не длинным текстом.
- Фильтры должны открываться снизу как мобильная панель.
- Телефонные номера должны быть кликабельными.
- Адрес и координаты должны открываться в картах.
- Ошибки данных показывать спокойно, как предупреждения, без блокировки просмотра.

## 12. Открытые вопросы

1. Кто будет основным пользователем MVP: региональные управляющие, территориальные менеджеры или центральный офис?
2. Нужна ли авторизация через корпоративную учетную запись?
3. Должны ли пользователи видеть все объекты или только свои регионы/НПО?
4. Является ли `cls_AZS` главным источником, а `data.csv` производной выгрузкой?
5. Нужно ли приложение только для просмотра или пользователи должны редактировать/комментировать данные?
6. Какие аналитические цифры первыми добавляем после MVP: продажи, НТУ, т/сут, сервисы, реконструкции, качество данных?
7. Можно ли использовать внешние карты, или нужна корпоративная/встроенная ГИС?
8. Как часто обновляются исходные файлы: ежедневно, еженедельно, ежемесячно?
9. Нужно ли хранить историю изменений между версиями файлов?
10. Какие поля считаются конфиденциальными: телефоны, ФИО, паспорт АЗС, адреса, бизнес-показатели?

## 13. Рекомендуемый следующий шаг

Согласовать MVP-состав экранов:

1. Главный экран: список + поиск + фильтры.
2. Карта объектов.
3. Карточка объекта.
4. Избранное.
5. Экран качества данных / объектов без координат.

После этого можно делать прототип экранов и проектировать backend-модель импорта данных.
